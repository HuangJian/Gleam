# 章十二 · 架构投影

> 承光之器可易，光之所依不可移。

## 一、架构是产品模型的投影

前十一章完成了从认知原理到产品概念的完整推导。
    从理解为何需要留存，
    到什么是值得留存的共鸣，
    再到留存的结构、时间中的演化、轨迹的形成、回望与表达，
    以及支撑这一切的 Repository、Recall 和 AI ——
这些内容共同构成了《Gleam · 拾光》的产品模型：
    一个以理解成长为核心、以 gleam 为基本单元、以时间为主轴的认知建模系统。

然而，产品模型本身还不是一个可运行的软件。
它描述了系统应该是什么，但尚未回答系统应该以什么样的技术结构来实现。
在产品模型与技术实现之间，需要一座桥梁 —— 系统架构。

系统架构的职责是：
    将产品模型中的核心概念和基本原则，
    映射为一组可执行的技术决策，
    使得软件在长期运行中能够始终表达同一个产品理念。

在这个意义上，架构是产品模型在实现空间中的投影。
每一个架构决策
    —— 无论是模块划分、数据存储、通信方式还是部署策略
    —— 都应当能够回答同一个问题：它如何帮助系统更忠实地实现产品模型所定义的目标？

如果一项技术选择不能回答这个问题，那么无论它在技术上多么先进，它都无缘成为这个系统的一部分。

## 二、核心模块及其边界

基于产品模型的职责划分，系统在逻辑上可以被分为若干个核心模块，每个模块承担产品模型中一个明确的职责。

Repository 是整个系统中最稳定的模块。
    它负责 gleam、Thought、Source 和所有元数据的持久化存储，是所有其他模块读取和写入数据的唯一权威来源。
    它的职责是保持数据的完整性、一致性和长期可访问性，不参与任何业务逻辑或计算任务。
Repository 模块的设计受第九章所定义的三条原则约束 ——
    统一性保证所有 gleam 属于同一个空间，
    不可变性保证历史记录不被覆盖，
    独立性保证 Repository 不强绑定于任何外部服务。

Capture 模块负责处理 gleam 的创建和写入。
当用户在阅读或思考过程中形成共鸣并决定拾取时，
    Capture 模块接收用户输入的 Thought 和 Source，
    生成时间戳和唯一标识，
    将它们封装为一个完整的 gleam，
    并写入 Repository。
这个模块的职责边界十分清晰：它只负责写入新的 gleam，不涉及任何读取、分析或关联操作。
这一边界保证了第八章所描述的"拾取应当轻过触碰"
    —— Capture 模块的工作足够简单，因此足够快速，不会打断用户当前的专注状态。

Timeline 模块负责 gleam 之间的关联和轨迹的生成。
它不制造新的数据实体，
    只通过读取 Repository 中的 gleam 及其元数据，
    根据时间和语义两个维度计算 gleam 之间的关系，
    并生成可供用户浏览的 Timeline 视图。
这个模块的输入是 Repository 中的 gleam 集合，输出是 Timeline 的结构化视图 ——
    它本身不存储数据，所有输出都可以在 Repository 发生变化时重新计算。
这种设计使 Timeline 始终反映 Repository 的最新状态，而不需要额外的同步机制。

Recall 模块负责在用户处于特定语境时，从 Repository 中检索相关的 gleam 和 Timeline。
它接收当前的语境信息（用户正在阅读的内容、最近浏览的 gleam、当前 Timeline 位置等），
    通过语义匹配和关联分析，
    识别出可能与当前语境相关的历史 gleam，
    并以非侵入的方式呈现给用户。
Recall 模块的设计与第十章描述的触发条件一致 ——
    它的输出基于当前语境，而非固定时间表，
    于是它只被动响应，不主动推送。

Intelligence 模块是系统中最"薄"的模块。
它只是一个不包含任何业务逻辑的适配层，
    将外部 AI（语义理解、关联分析、聚类、摘要生成等）接入系统，
    并将它们的结果提供给 Timeline、Recall 和其他需要语义分析能力的模块使用。
Intelligence 模块的设计遵循第十一章的两个原则：它不创建新的 gleam，所有输出都是可替换的派生数据。
这个模块的薄化设计，使得系统对特定模型或服务商的依赖降到最低 ——
    如果外部能力发生变化，只需要替换 Intelligence 模块的适配层，不会影响系统的其他部分。

## 三、依赖方向：从稳定到可变

以上五个模块的职责边界清晰后，接下来还要建立它们之间的依赖关系原则：
    依赖方向应当始终从可变模块指向稳定模块。

稳定模块的定义是：它的行为由产品模型直接决定，不随技术选型或外部条件的变化而变化。
在这个系统中，Repository 是最稳定的模块 ——
    它的数据结构和访问方式由第三章的 gleam 定义和第九章的三条原则所决定，
    无论外部技术如何变化，这些核心定义都不应改变。

可变模块的定义是：它的实现方式可能随技术发展、用户需求或外部环境的变化而调整。
    Capture 模块的交互方式可能随着新设备或新交互范式的出现而变化，
    Timeline 模块的关联算法可能随着语义理解技术的进步而升级，
    Recall 模块的检索策略可能随着用户使用习惯的反馈而优化，
    Intelligence 模块的模型选型可能随着行业的演化而频繁更替。

依赖方向从可变指向稳定的含义是：可变模块可以依赖稳定模块 ——
    Capture 需要向 Repository 写入数据，
    Timeline 需要从 Repository 读取数据，
    Recall 需要查询 Repository 中的数据，
    Intelligence 需要读取 Repository 中的 gleam 内容来做语义分析。
但稳定模块不依赖任何可变模块 ——
    Repository 不知道也不关心是谁在写入数据或谁在读取数据，它只负责数据的持久化和完整性。

这种依赖方向的设计确保了系统的核心稳定性。
当某个可变模块需要修改或替换时，其他模块不会受到影响，因为它们的依赖关系都指向同一个稳定的核心 ——
    Repository 是系统长期演化的锚点，其他模块围绕它变化，而它本身始终不变。

## 四、架构即约束

系统架构描述模块和依赖关系之余，更是一种约束的声明。

一个明确定义的架构，实际上是在说：
某些变化是系统可以接受的（例如
    更新 Timeline 的关联算法、
    替换 Intelligence 的模型提供商、
    优化 Recall 的检索策略），
而某些变化是系统不允许的（例如
    修改 gleam 的数据结构而破坏旧记录的完整性、
    让外部服务成为数据的唯一持有者、
    允许 AI 生成的内容作为 gleam 进入 Repository）。

这些约束是产品模型对系统实现施加的要求。
它们使系统在应对技术演化和需求变化时，始终保持与产品模型的一致，而不至于在实现层面的修改中逐渐偏离原有的设计目标。

在这个意义上，系统架构是产品模型在技术空间中的守护者。
它通过定义哪些模块负责什么、哪些依赖是允许的、哪些变化是可以接受的，
    来确保系统在长期运行中始终忠实于前十一章所建立的产品理念。
当技术栈更新时，当用户需求变化时，当新的交互方式出现时，
    架构的约束使这些变化只发生在系统的外围，而不会侵蚀核心。

----

被架构原则小心保存下的 gleam，将得以穿越技术的更迭，以完整面貌静候每一次重逢。
