# 章九 · 记忆居所

> 微光需要一处不为风雨飘摇的居所，方能安心沉淀绵延。

## 一、留存需要一处稳定的空间

前八章已经建立了一套完整的理解留存与演化模型。
从理解为何需要留存，到何为值得留存，再到留存的结构、时间中的历程、轨迹的形成、回望与表达 ——
    这些内容共同构成了一个完整的产品模型。
这个模型回答了"系统应该是什么"的问题。

然而，所有被留存的 gleam、Timeline 和 Milestone，都需要一个实际存在的空间来安置它们。
这个空间不能只是一个抽象概念，它必须在物理层面或逻辑层面具有真实的载体 ——
    能够存储数据、维护结构、响应访问，并在数十年间保持稳定可用。
第一章所描述的"跨越时间的留存"，只有在这个空间真实存在的前提下才能实现。

因此，在产品模型确立之后，系统必须定义一种具体的机制来承载它。这个承载机制被称为 Repository。

Repository 在概念上是一个统一的、可信的、长期稳定的记忆空间，
    所有 gleam、Thought、Source、Timeline 和 Milestone 都保存在其中，并能够被持续访问、检索和组织。
在实现层面，它可以是本地文件系统上的一个数据库，也可以是云端的一个存储服务，或者两者的结合体。
但无论实现方式如何变化，Repository 在概念层面的定义应始终保持一致：
    它是理解留存的物理基础，
    是 gleam 在时间中持续存在的实际载体，
    也是系统实现长期记忆的唯一可信来源。

## 二、统一性是连续性的前提

Repository 的第一个基本原则是：所有 gleam 必须属于同一个空间。

这一原则的推导直接来自第五章所建立的 Timeline 概念。
Timeline 是一条跨越时间的认知轨迹，它的连续性依赖于所有相关 gleam 都能够被定位在同一参照系中。
如果一条 gleam 保存在浏览器扩展的本地存储中，
    另一条 gleam 保存在移动应用的私有数据库里，
    第三条 gleam 保存在云端服务的某个项目中，
那么即使它们在语义上高度相关，系统也无法将它们连接为一条完整的 Timeline ——
    因为它们分布在相互隔离的空间中，缺少统一的身份标识和访问机制。

因此，无论 gleam 从哪一个入口进入系统 ——
    浏览器扩展、
    桌面应用、
    移动设备、
    命令行工具，
    还是未来可能出现的新交互方式 ——
它们最终都必须汇聚到同一个 Repository 中。
这个统一性保证了 Timeline 的连续性不会被输入方式所切割，也保证了用户的思考轨迹不会因为设备更换或软件升级而断裂。

统一性还意味着，Repository 是系统中唯一的数据权威来源。
所有 gleam 的创建、访问、检索和组织，都以 Repository 中的数据为准；
所有 Timeline 的生成和 Milestone 的识别，都以 Repository 中存储的 gleam 及其元数据为基础。
任何外部服务或缓存都只是 Repository 的辅助和镜像，而不应成为数据的最终持有者。

## 三、不可变性作为历史完整性的保障

Repository 的第二个基本原则是：一旦 gleam 被写入，其核心内容便封存在那一刻，不再被修改。

这条原则来自第一章所建立的"成长需要回望"这一核心命题。
回望要求用户能够在不同的时间点重新访问同一个 gleam，并观察到自己的理解在时间中的变化。
如果 gleam 的内容可以被修改，那么回望时就无法确定"当时的理解"是什么 ——
    用户看到的可能是被修订后的版本，那段理解形成时的真实面貌却已无处可寻。

因此，Repository 必须保证每一条 gleam 都具有历史完整性。
它在写入时的内容、时间戳和 Source，都应当保持不变。
如果用户对某个主题形成了新的理解，它应当作为一条新的 gleam 被写入，不应覆盖原有的 gleam。
这样，旧的 gleam 保留了"过去的理解"，新的 gleam 记录了"现在的理解"，两者在 Timeline 上共存，共同构成一条完整的演化轨迹。

这条原则在实现层面可能带来一些挑战 ——
    例如用户可能会想要修正拼写错误、
    补充遗漏的信息，
    或调整 Source 的引用格式。
这些修改需求可以借由允许编辑元数据（如标签、备注）而保留核心 Thought 不变来解决，也可以借由创建勘误式的新 gleam 来表达修正。
关键在于，原始 gleam 作为历史记录的身份必须得到保护。

## 四、独立性作为长期可访问性的保障

Repository 的第三个基本原则是：它应当独立于任何外部服务而存在。

这条原则的推导来自一个实际考量：系统可能运行很多年，具体的软件形态、技术栈和部署方式却会在此过程中几经变迁。
今天的云服务可能明天便停止维护，当前的 AI Provider 可能下个月就更换，而 Repository 中存储的 gleam 需要在所有这些变化中保持可访问性。
如果 Repository 完全依赖于某一个外部服务，那么当该服务不可用时，用户将无法访问自己的理解历史。

因此，Repository 必须在结构上保持独立。
    所有 gleam、Thought、Source 和元数据都以标准化的格式存储，不依赖特定数据库引擎或特定厂商的 API；
    同步和备份机制应当是可选的、可替代的，而不是强制绑定的；
    未来即使整个软件停止维护，用户也应该能够通过其他工具打开和读取 Repository 中的数据。

独立性还意味着，用户对 Repository 拥有最终的控制权。
存储位置由用户选择，数据导出格式公开，用户可以在任何时候将所有数据迁移到其他系统。
这种控制权既是一种信任的体现，也是"思考属于用户而非软件"这一核心理念在数据层的实现。

## 五、Repository 作为长期记忆的基础设施

当统一性、不可变性和独立性这三个原则同时成立时，Repository 便具备了成为长期记忆基础设施的资格。

它既是 gleam 的存储容器，也是理解成长的支撑结构。
    统一性使 Timeline 能够在不同的输入方式之间保持连续；
    不可变性使回望能够基于真实的历史记录；
    独立性使 Repository 能够在软件形态不断变化的情况下依然保持可访问。
这三个原则共同构成了一套保障，使 gleam 能够在数十年间保持完整、一致和可用。

从产品设计的角度来看，Repository 还承担着一种更底层的角色：
    它将理解留存从"使用软件的功能"转变为"拥有数据的资产"。
当用户使用一个常规软件时，他的数据通常被视为软件运行过程中的副产品；
而在《Gleam · 拾光》的设计中，Repository 是独立于软件本身的存在，软件只是访问和操作 Repository 的工具之一。
这种关系使 Repository 拥有了超越软件生命周期的持久性 ——
    即使当前版本的软件停止开发，Repository 中的数据仍然是用户自己的。
