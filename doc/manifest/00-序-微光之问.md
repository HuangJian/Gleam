# 序 · 微光之问

这本手册的起点，是一个简单到几乎不言自明的问题：一个人如何知道自己真的理解了一件事？

大多数人并不需要回答这个问题就能正常生活。
日常经验告诉我们，理解是一种内在的确定感 ——
    当某段内容与自己的经验发生共鸣时，我们知道自己理解了，不需要外部验证。
这种感觉是可靠的，也是即时发生的，它驱动着日常的阅读、学习和思考。

然而，这种即时感有一个隐蔽的弱点：它只在当下有效。
三个月后，当同样的内容再次出现时，你可能会发现自己曾经拥有的那种确定感已经模糊了 ——
    你知道自己理解过，却再难进入当时的状态。
这无关记性好坏，根源在于理解本身的时间性：理解需要时间才能成熟，同时也会在时间中褪去。

因此，与其纠结"如何确认理解"，不如追问"如何让理解跨越时间"。

这本手册记录了一个试图解决这个问题的设计过程。
这是一部从认知起点出发、逐层推导出完整系统形态的架构手册，与产品功能介绍、设计随笔皆不相同。
这部手册不会给出现成的结论。
它只做一件事：带着读者一步步推导 ——
    每一章都从上一章的结论出发，使读者知道系统最终是什么样之外，更理解它为什么必须是这个样子。

手册分为四个部分。
第一部分建立存在的前提：理解为什么需要被留存，留存什么，以及留存的结构是什么。
第二部分描述理解的成长过程：从共鸣到拾取，从重遇到沉淀，从回望到表达。
第三部分将认知模型转化为系统设计：交互应当如何保护理解，Repository 如何承载记忆，Recall 如何让过去重新参与现在，AI 如何以克制的姿态介入。
第四部分将设计转化为可运行的系统：架构如何成为产品模型的投影，gleamd 如何在时间中守护秩序，以及系统如何在不断变化中保持其核心。

阅读这本手册，不必拘泥于从头到尾的线性顺序。
更好的方式，是跟随每一个"为什么"逐步走进系统内部。
当你读完最后一章时，希望你不仅了解到系统有哪些功能，更是理解了那些看似微小的设计决策，
    如何从"理解需要时间"这一朴素起点开始，经由层层推导，最终汇聚为一个完整而自洽的系统形态。
